Why use const with a reference return? Regular return types are rvalues(cannot be accessed by address), but a reference type could be assigned a value. Add const could protect from this.

Suppose the actual argument type doesn’t match the reference parameter type but can be converted to the reference type. Then the program creates a temporary variable of the correct type, initializes it to the converted value, and passes a reference to the tempo- rary variable

Explicit specialization && explicit instantiation(generate immediately)

Specialized function definition, for already defined templates. eg:

instantiation and specialization.

Template is not a function definition, but a plan to generate right definition.

  • implicit instantiation: generate function when notice the program use one with right parameters.
  • explict instantiation: instruct the compiler to generate a certain instantiation. eg: Swap<int>();

404 note: If use implicit instantiation, all .o files had better to be fed to the compiler at the same time, to let compiler know every possible usage of a certain function? So implicit instantiation is not good for partly compilation?

Caution: It is an error to try to use both an explicit instantiation and an explicit specialization for the same type(s) in the same file, or, more generally, the same translation unit.

About c++ partial ordering rules

I hope I would never encounter this.

The decltype keyword: to solve unknown type like x + y in template.

decltype(expression) var; 4 stages.

  • if expression is unparenthesized identifier, var is the same type.
  • a function call - same type as return value
  • a lvalue: a reference to the expression type. (Not very clear about this-parentheses should be added explicitly as an identifier?)
  • else: same type as expression.

Another problem of unsolved type in function templating: about the return value.
